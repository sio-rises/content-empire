2:T1a4e,<h2>はじめに：テストコード自動生成の価値</h2>
<p>テスト駆動開発（TDD）は品質向上に有効だが、テストコードを書く工数は無視できない。OpenCodeはLLMを活用し、自然言語の仕様から自動でテストコードを生成するツールだ。本記事では、OpenCodeを用いてJestテストコードを自動生成し、TDDサイクルを加速する方法を解説する。</p>
<h2>OpenCodeの基本セットアップ</h2>
<p>まずはOpenCodeを導入する。Node.js環境が必要だ。</p>
<pre><code class="language-bash">&lt;h1&gt;グローバルインストール&lt;/h1&gt;
npm install -g @opencode/cli
<p>&lt;h1&gt;プロジェクト初期化&lt;/h1&gt;
mkdir my-project &amp;&amp; cd my-project
npm init -y
</code></pre></p>
<p>次に、テストフレームワークとしてJestをインストールする。</p>
<pre><code class="language-bash">npm install --save-dev jest
</code></pre>
<p>OpenCodeの設定ファイル<code>.opencode.yml</code>をプロジェクトルートに作成する。</p>
<pre><code class="language-yaml">&lt;h1&gt;.opencode.yml&lt;/h1&gt;
testFramework: jest
testPattern: "**/*.test.js"
sourcePattern: "src/**/*.js"
</code></pre>
<h2>テストコード自動生成の基本フロー</h2>
<p>OpenCodeは「Specification（仕様）」からテストコードを生成する。Specificationは自然言語で記述し、Markdownファイルとして管理する。</p>
<h3>ステップ1：Specificationファイルを作成</h3>
<p><code>specs/calculator.md</code>を作成する。</p>
<pre><code class="language-markdown">&lt;h1&gt;Calculator&lt;/h1&gt;
<p>&lt;h2&gt;add&lt;/h2&gt;
&lt;li&gt;2つの数値を受け取り、その和を返す&lt;/li&gt;
&lt;li&gt;引数が数値でない場合はエラーを投げる&lt;/li&gt;</p>
<p>&lt;h2&gt;subtract&lt;/h2&gt;
&lt;li&gt;2つの数値を受け取り、差を返す&lt;/li&gt;
&lt;li&gt;引数が数値でない場合はエラーを投げる&lt;/li&gt;
</code></pre></p>
<h3>ステップ2：テストコードを生成</h3>
<pre><code class="language-bash">opcode generate specs/calculator.md --output tests/calculator.test.js
</code></pre>
<p>生成される<code>tests/calculator.test.js</code>は以下のようになる。</p>
<pre><code class="language-javascript">const { add, subtract } = require('../src/calculator');
<p>describe('Calculator', () =&gt; {
  describe('add', () =&gt; {
    test('returns sum of two numbers', () =&gt; {
      expect(add(1, 2)).toBe(3);
      expect(add(-1, 1)).toBe(0);
      expect(add(0, 0)).toBe(0);
    });</p>
<p>test('throws error if arguments are not numbers', () =&gt; {
      expect(() =&gt; add('a', 1)).toThrow();
      expect(() =&gt; add(1, null)).toThrow();
    });
  });</p>
<p>describe('subtract', () =&gt; {
    test('returns difference of two numbers', () =&gt; {
      expect(subtract(5, 3)).toBe(2);
      expect(subtract(0, 1)).toBe(-1);
    });</p>
<p>test('throws error if arguments are not numbers', () =&gt; {
      expect(() =&gt; subtract('a', 1)).toThrow();
      expect(() =&gt; subtract(1, undefined)).toThrow();
    });
  });
});
</code></pre></p>
<h3>ステップ3：実装コードを作成しテストを実行</h3>
<pre><code class="language-bash">&lt;h1&gt;実装コードのひな形を作成&lt;/h1&gt;
mkdir src
touch src/calculator.js
<p>&lt;h1&gt;テストを実行（最初は失敗する）&lt;/h1&gt;
npm test
</code></pre></p>
<p>実装を書いてテストを通す。</p>
<pre><code class="language-javascript">// src/calculator.js
function add(a, b) {
  if (typeof a !== 'number' || typeof b !== 'number') {
    throw new Error('Arguments must be numbers');
  }
  return a + b;
}
<p>function subtract(a, b) {
  if (typeof a !== 'number' || typeof b !== 'number') {
    throw new Error('Arguments must be numbers');
  }
  return a - b;
}</p>
<p>module.exports = { add, subtract };
</code></pre></p>
<h2>TDDワークフローへの統合</h2>
<p>TDDのサイクル「Red-Green-Refactor」にOpenCodeを組み込む。</p>
<li><strong>Red</strong>: Specificationを書き、テストコードを生成して実行（失敗確認）</li>
<li><strong>Green</strong>: 実装コードを書き、テストを通す</li>
<li><strong>Refactor</strong>: コードを改善し、テストが通ることを確認</li>
<p>SpecificationファイルをGit管理することで、仕様の変更履歴も追跡できる。</p>
<h2>高度な使い方：複数ファイルと依存関係</h2>
<p>大規模プロジェクトでは、Specificationを分割して管理する。OpenCodeは<code>@import</code>ディレクティブをサポートしている。</p>
<pre><code class="language-markdown">&lt;h1&gt;specs/user-service.md&lt;/h1&gt;
@import specs/common/errors.md
<p>&lt;h2&gt;createUser&lt;/h2&gt;
&lt;li&gt;有効なメールアドレスとパスワードを受け取り、ユーザーを作成する&lt;/li&gt;
&lt;li&gt;メールアドレスが既存の場合はエラーを投げる&lt;/li&gt;
</code></pre></p>
<p>生成コマンドで複数ファイルを指定できる。</p>
<pre><code class="language-bash">opcode generate specs/ --output tests/
</code></pre>
<h2>品質を高めるポイント</h2>
<li><strong>Specificationを具体的に書く</strong>: 「正常系」「異常系」を網羅し、境界値も明記する</li>
<li><strong>生成後にレビューする</strong>: 自動生成されたテストコードは必ず人間が確認し、不要なテストや誤ったアサーションを修正する</li>
<li><strong>モックを活用する</strong>: OpenCodeはモックの自動生成も可能。Specificationに外部依存を明記すれば、Jestのモック関数を生成してくれる</li>
<li><strong>継続的に更新する</strong>: 仕様変更時はSpecificationを更新し、テストコードを再生成。手動修正との差分を管理する</li>

<h2>まとめ</h2>
<p>OpenCodeを使えば、Specificationからテストコードを自動生成でき、TDDの導入障壁を下げられる。ただし、生成されたコードを盲信せず、レビューと修正を怠らないことが重要だ。Specificationを「生きたドキュメント」として育てることで、プロジェクトの品質と開発速度を両立できる。</p>
